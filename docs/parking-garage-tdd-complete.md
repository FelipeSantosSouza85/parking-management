## 1. Context

This system represents a backend service responsible for managing parking garage operations, including vehicle entry and exit processing, parking spot allocation, and revenue calculation.

The service integrates with an external garage simulator that emits parking events through a webhook interface. These events represent the lifecycle of a vehicle within the parking garage, including entry, parking confirmation, and exit.

The system is responsible for maintaining the internal state of the garage, ensuring spot availability consistency, applying business rules for pricing and occupancy, and calculating parking revenue.

The application exposes a REST API that allows querying the revenue generated by the parking garage based on date and sector.

### Stakeholders

- Estapar Engineering Team

### Main Objectives

- Process parking events received from the external garage simulator
- Maintain the consistent state of the parking garage
- Apply pricing and occupancy business rules
- Calculate parking revenue
- Provide an API endpoint to query revenue by sector and date

## 2. Problem Statement & Motivation

Parking garages require reliable backend systems capable of processing vehicle lifecycle events in real time while maintaining a consistent view of parking spot availability and calculating accurate parking revenue.

The system must react to events produced by an external simulator and ensure that the internal state of the parking garage remains consistent at all times.

### Technical Challenges

1. **Event-driven state management**  
   The system must react to events such as `ENTRY`, `PARKED`, and `EXIT` and correctly update the internal state of parking sessions and parking spots.

2. **Concurrency control**  
   Multiple events may arrive simultaneously. The system must ensure that:
   - A parking spot cannot be occupied by more than one vehicle
   - The garage capacity constraints are respected

3. **Pricing rules application**  
   Parking fees must be calculated based on:
   - Duration of stay
   - Sector base price
   - Dynamic price adjustments depending on occupancy

4. **Integration with an external simulator**  
   The system depends on webhook events generated by an external garage simulator and must reliably process these events.

### Impact if the Problem is Not Properly Solved

If the system does not correctly handle these responsibilities, several operational problems may occur:

- Parking spot state inconsistencies
- Overbooking situations
- Incorrect revenue calculation
- Operational failures within the parking garage system

### Why the Solution Matters

Providing a reliable backend system ensures:

- Operational reliability for parking garage management
- Data integrity across parking sessions and spots
- Accurate financial reporting and revenue tracking
- A scalable architecture that can evolve to support future integrations

### Architectural Style

The solution will be implemented as a **modular monolith**, organized by business domain.

Each module contains clear internal separation of responsibilities across:
- `api`
- `application`
- `domain`
- `infrastructure`

This approach provides:
- Clear business-oriented organization
- High cohesion within each module
- Reduced coupling between functional areas
- Simpler operational model than microservices
- Good maintainability and testability for the scope of the challenge

### Internal Communication Strategy

Internal module communication will be implemented through **concrete application services** to keep the modular monolith simple, explicit, and easy to maintain.

Interfaces will be introduced only at meaningful architectural boundaries, especially for **external integrations** where abstraction provides real value.

### External Integration Strategy

For external integrations, **declarative REST client interfaces** will be used directly, since the framework provides the runtime implementation and no additional adapter logic is required for this solution.

### Separação Domínio vs Persistência

As entidades de domínio são **POJOs puros**, sem anotações JPA. A persistência fica isolada em:

- **domain/**: `GarageSector`, `ParkingSpot`, `GarageOccupancy` — regras de negócio, invariantes, comportamentos
- **application/port/**: interfaces (`GarageSectorPort`, `ParkingSpotPort`, `GarageOccupancyPort`) — contratos de persistência
- **infrastructure/persistence/entity/**: entidades JPA (`*Entity`) — mapeamento para tabelas
- **infrastructure/persistence/mapper/**: conversão entre entidades JPA e objetos de domínio
- **infrastructure/persistence/repository/adapter/**: implementações dos Ports usando JpaRepository

### Layer Responsibilities

#### `api`
- Expose HTTP endpoints
- Receive request DTOs
- Delegate to application services
- Avoid business logic

#### `application`
- Orchestrate use cases
- Define transaction boundaries
- Coordinate domain operations

#### `domain`
- Hold core business entities

- Represent business invariants and calculations

#### `infrastructure`
- Implement persistence access via **Adapters** that implement **Ports**
- Host **JPA entities** (`*Entity`) and **Mappers** (Entity ↔ Domain)

- `updatedAt`

### Why `GarageOccupancy` Exists

A dedicated global occupancy record is maintained to simplify concurrency control for vehicle entry.

This record is used exclusively for **global entry capacity control**, since vehicle entry happens through a single gate.

Sector-specific occupancy will not be materialized as a separate aggregate in V1, because:
- The physical allocation is already controlled by `ParkingSpot`
- A separate sector occupancy aggregate would introduce extra write synchronization complexity
- The current challenge scope does not justify duplicating this state

### Session Statuses

- `ENTERED`
- `PARKED`
- `EXITED`
- `REJECTED`

### Relationships

- One `GarageSector` has many `ParkingSpot`
- One `ParkingSession` may be associated with one `ParkingSpot`
- One `ParkingSession` may be associated with one `GarageSector` after `PARKED`

### Domain Rules

1. `ParkingSpot` represents the physical state of the spot
2. `ParkingSession` represents the lifecycle of the vehicle
3. The dynamic pricing adjustment is captured at `ENTRY`
4. The sector base price is determined when the real spot is known at `PARKED`
5. The final charge is calculated at `EXIT`
6. Charges are only applied when a vehicle is effectively parked in a spot
7. If a vehicle enters and exits without a `PARKED` event, the session is closed with zero charge

## 8. Event Flow / Lifecycle

### `ENTRY`

When the system receives an `ENTRY` event:

1. Validate the payload
2. Check whether an active session already exists for the license plate
3. Load `GarageOccupancy` with a transactional lock
4. Validate whether the garage is full
5. Calculate the dynamic pricing adjustment based on global occupancy at that moment
6. Create a `ParkingSession` with status `ENTERED`
7. Increment `occupiedCount` in `GarageOccupancy`
8. Return success

**Captured data**
- `occupancyRateAtEntry`
- `priceAdjustmentRateAtEntry`

**Notes**
- At this moment, the spot is still unknown
- At this moment, the sector is still unknown

### `PARKED`

When the system receives a `PARKED` event:

1. Validate the payload
2. Load the active session for the license plate with a lock
3. Find the parking spot by `lat/lng`
4. Validate that the spot exists
5. Validate that the spot is free
6. Mark the spot as occupied
7. Associate `spot` and `sector` to the session
8. Set `parkedTime`
9. Calculate and store `hourlyPriceApplied`
10. Change session status to `PARKED`
11. Return success

**Notes**
- The real sector is defined here from the physical spot
- The final hourly price can only be determined at this stage

### `EXIT`

When the system receives an `EXIT` event:

1. Validate the payload
2. Load the active session for the license plate with a lock
3. Set the exit time
4. If the session is `PARKED`:
   - Calculate duration
   - Calculate final charge
   - Release the associated `ParkingSpot`
5. If the session is `ENTERED` and has no associated spot:
   - Close the session with zero charge
6. Mark session as `EXITED`
7. Decrement `occupiedCount` in `GarageOccupancy`
8. Return success

### Special Case: Entry Without Parking

A vehicle may enter the garage and leave without ever parking in a spot.

Valid lifecycle:
- `ENTRY`
- `EXIT`

In this case:
- No `ParkingSpot` is assigned
- No `GarageSector` is assigned
- No parking fee is charged
- The session transitions directly from `ENTERED` to `EXITED`

This reflects the real-world scenario where a driver enters the garage, decides not to park, and leaves without using a parking space.

## 9. Pricing Rules

### Free Tolerance

- Up to **30 minutes**: free
- More than 30 minutes: charged by hour
- Hours are rounded **up**
- The first hour is charged once the 30-minute tolerance is exceeded

### Examples

- `20 min` → `0`
- `31 min` → `1 hour`
- `61 min` → `2 hours`

### Dynamic Pricing Rule

The occupancy-based adjustment is captured at `ENTRY`, using **global garage occupancy**, because:
- Entry happens through a single gate
- The sector is still unknown at this point

Adjustment table:

- `< 25%` → `-0.10`
- `<= 50%` → `0.00`
- `<= 75%` → `0.10`
- `<= 100%` → `0.25`

### Final Hourly Price Formula

The final hourly price is determined once the sector is known during `PARKED`.

```text
hourlyPriceApplied = sector.basePrice * (1 + priceAdjustmentRateAtEntry)
```

### Charging Rule

Charging is only applied if:
- The vehicle was effectively parked
- The sector is known
- `hourlyPriceApplied` was determined

If the vehicle entered and exited without parking:
- `chargedAmount = 0`

## 10. Concurrency & Transaction Strategy

### Transaction Principle

Each event (`ENTRY`, `PARKED`, `EXIT`) will be processed inside a **single database transaction**.

This guarantees:
- Atomicity
- Rollback on failure
- Consistent state transitions

### Critical Resources

The following resources require concurrency protection:

- `GarageOccupancy`
- `ParkingSpot`
- `ParkingSession`

### Locking Strategy

The solution will use **pessimistic locking** for critical resources.

This choice is appropriate because:
- The domain involves scarce resources (garage capacity and parking spots)
- Conflicts are expected and meaningful
- Deterministic behavior is preferred over retry-based conflict handling

### Entry Concurrency

For `ENTRY`:
- `GarageOccupancy` is loaded using pessimistic write lock
- Capacity is validated under lock
- Session creation and occupancy increment happen within the same transaction

This prevents overbooking.

### Parked Concurrency

For `PARKED`:
- `ParkingSession` is loaded with lock
- `ParkingSpot` is loaded with lock
- Spot occupancy is updated transactionally

This prevents two vehicles from occupying the same spot.

### Exit Concurrency

For `EXIT`:
- `ParkingSession` is loaded with lock
- `ParkingSpot` is loaded with lock when applicable
- `GarageOccupancy` is updated transactionally

This prevents duplicated exit processing and inconsistent release behavior.

### Lock Order

To reduce deadlock risk, resources will be locked in a consistent order:

1. `ParkingSession`
2. `ParkingSpot`
3. `GarageOccupancy`

### Idempotency Strategy

Idempotency in V1 will be based on **session state validation**, not on a dedicated processed-event table.

Examples:
- `ENTRY` for a plate that already has an active session → conflict
- `PARKED` for a session already parked → conflict
- `EXIT` for a session already exited → conflict

This keeps the solution simple while still protecting business state consistency.

Suggested request fields:
- `licensePlate`
- `entryTime`
- `exitTime`
- `lat`
- `lng`
- `eventType`

#### Response Strategy

The webhook endpoint will use explicit HTTP semantics:

- `200 OK` → event successfully processed
- `400 Bad Request` → invalid payload
- `404 Not Found` → required resource not found
- `409 Conflict` → business rule or invalid state transition
- `500 Internal Server Error` → unexpected failure

#### Error Response Format

```json
{
  "code": "ERROR_CODE",
  "message": "Human readable description",
  "timestamp": "2026-03-11T18:30:00Z"

```

#### Example Error Cases

- `GARAGE_FULL`
- `ACTIVE_SESSION_NOT_FOUND`
- `SPOT_NOT_FOUND`
- `SPOT_ALREADY_OCCUPIED`

#### Responsibility

Returns aggregated revenue by date and sector.

#### Endpoint Format

```http
GET /revenue?date=2025-01-01&sector=A
```

#### Query Parameters

- `date` (required)
- `sector` (required)

#### Response Example


#### Aggregation Rule

Revenue is calculated by summing:
- `charged_amount`
- from sessions with status `EXITED`
- filtered by:
  - `sector`

## 13. Sequence of Major Flows

### Startup / Garage Sync

1. Application starts
2. `GarageSynchronizationService` calls `GarageSimulatorClient`
3. Garage sectors are persisted or updated
4. Parking spots are persisted or updated
5. `GarageOccupancy.totalCapacity` is initialized from sector capacities
6. `GarageOccupancy.occupiedCount` is initialized based on simulator snapshot or zeroed according to chosen sync strategy

### Webhook Event Dispatch

1. `WebhookController` receives request
2. Payload is validated
3. `WebhookEventDispatcher` routes by `eventType`
4. Proper parking application service is invoked
5. Response is returned with success or standardized error

### Revenue Query

1. `RevenueController` receives date and sector
2. `RevenueQueryService` queries aggregated charged sessions
3. `RevenueResponse` is returned in BRL

## 16. Failure Handling & Rollback Strategy

### Transactional Rollback

Each event is processed inside a single database transaction.

If any step fails:
- The transaction is rolled back
- No partial changes are persisted

This is the main consistency guarantee of the solution.

### Business Failures

Business rule failures do not produce partial side effects.

Examples:
- Garage full
- Spot already occupied
- Missing active session
- Invalid session transition

In these cases:
- Nothing is persisted
- The API returns a proper error response
- The system remains consistent

### Unexpected Failures

Unexpected exceptions will:
- Roll back the transaction
- Return `500 Internal Server Error`
- Be logged with sufficient context

### Duplicate Events

Duplicate events are handled through session-state validation in V1.

### Out-of-Order Events

Examples:
- `PARKED` without active session → error
- `EXIT` without active session → error
- `EXIT` after `ENTRY` but without `PARKED` → valid, zero charge

### Operational Rollback Scope

This solution does not include:
- Deployment rollback automation
- Event compensation flows
- Message replay infrastructure

Rollback scope in V1 is limited to:
- Database transaction rollback
- Reprocessing through normal event flow if needed
